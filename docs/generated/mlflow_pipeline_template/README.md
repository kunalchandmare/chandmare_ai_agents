# mlflow_pipeline_template вЂ” Documentation

## Overview

`mlflow-pipeline-template` is a pip-installable CLI tool that generates a fully structured MLflow + Hydra pipeline project from two user-provided YAML files.

---

## Installation

```bash
pip install chandmare-mlflow-pipeline-template
```

---

## Command

```bash
# Generate a project (config.yaml and pipeline.yaml in current directory)
mlflow-pipeline-template generate ./my_project

# Specify custom config file paths
mlflow-pipeline-template generate ./my_project --config my_config.yaml --pipeline my_pipeline.yaml
```

---

## Input Files

The user provides exactly **two YAML files**:

### 1. `config.yaml` вЂ” Project-level configuration

```yaml
project_name: "my_ml_project"
artifact_backend: "mlflow"       # mlflow | dvc | wandb
mlflow_version: "2.14.1"
# wandb_entity: "myteam"        # only if artifact_backend: wandb
# wandb_project: "my_project"   # only if artifact_backend: wandb
# dvc_remote: "s3://bucket"     # only if artifact_backend: dvc
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `project_name` | str | *(required)* | Project name, used as MLflow project name and folder name |
| `artifact_backend` | str | `"mlflow"` | Artifact versioning backend: `mlflow`, `dvc`, or `wandb` |
| `mlflow_version` | str | `"2.14.1"` | MLflow version to pin in generated environments |
| `wandb_entity` | str | `""` | W&B entity (only when `artifact_backend: wandb`) |
| `wandb_project` | str | `project_name` | W&B project name (only when `artifact_backend: wandb`) |
| `wandb_version` | str | `"0.17.0"` | W&B version (only when `artifact_backend: wandb`) |
| `dvc_remote` | str | `""` | DVC remote URL (only when `artifact_backend: dvc`) |

### 2. `pipeline.yaml` вЂ” Pipeline definition

```yaml
steps:
  clean:
    description: "Apply data cleaning"
    arguments:
      input_artifact:
        type: str
        required: true
        description: "Input artifact"
      min_price:
        type: float
        default: 10.0
        description: "Minimum price threshold"

components:
  download_data:
    description: "Generic artifact downloader"
    arguments:
      artifact_name:
        type: str
        required: true
        description: "Artifact to download"
```

#### Argument schema

Each argument under `arguments:` supports:

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | str | yes | Python type: `str`, `int`, `float`, `bool` |
| `default` | any | no | Default value. If absent, argument is required |
| `required` | bool | no | Explicitly mark as required (default: `false`) |
| `description` | str | yes | Help text for argparse and MLproject |

Arguments auto-propagate to:
- `run.py` в†’ `argparse.add_argument()` calls
- `MLproject` в†’ `parameters:` section

## Architecture: components/ vs src/

### `src/` вЂ” Project-Specific Steps

Steps defined under `steps:` in `pipeline.yaml`. They contain **domain logic** вЂ” know column names, thresholds, model architecture.

- вќЊ Not reusable across projects
- вњ… Contains the actual ML intelligence
- Examples: `clean`, `train`, `feature_eng`, `evaluate`

### `components/` вЂ” Reusable, Schema-Agnostic Blocks

Components defined under `components:` in `pipeline.yaml`. They are **generic** вЂ” work with any data without modification.

- вњ… Reusable across any project
- вќЊ Must NOT contain domain logic
- Examples: `download_data`, `train_val_test_split`, `test_model`

### Decision flowchart

```
Does the step know column names, data types, or business rules?
  YES в†’ put it under steps: (generates to src/)
  NO  в†’ Does it operate on artifacts as opaque files?
    YES в†’ put it under components: (generates to components/)
    NO  в†’ probably doesn't belong in the pipeline
```

## Artifact Backends

### MLflow (default)

Zero setup. Uses `mlflow.log_artifact()` and `mlflow.artifacts.download_artifacts()`.

### DVC (opt-in)

Git-like data versioning. Uses `dvc add`, `dvc push`, `dvc pull`. Best for large datasets shared across a team.

### W&B (opt-in)

Cloud artifact tracking with lineage UI. Uses `wandb.Artifact`. Requires account.

### Key files

| File | Purpose |
|---|---|
| `cli.py` | Parses `mlflow-pipeline-template generate` command, validates inputs, calls `generator.py` |
| `generator.py` | Reads config.yaml + pipeline.yaml, renders templates, writes output project |
| `_blueprints/step/run.py.jinja` | Template for `src/<step>/run.py` вЂ” argparse from pipeline.yaml arguments |
| `_blueprints/step/MLproject.jinja` | Template for `src/<step>/MLproject` вЂ” parameters from pipeline.yaml |
| `components/shared/artifact_utils/` | Utility files included in generated project |

## Adding steps to an existing project

1. Edit `pipeline.yaml` вЂ” add new step/component definition
2. Re-run: `mlflow-pipeline-template generate . --config config.yaml --pipeline pipeline.yaml`
3. Existing `run.py` files are **never overwritten** вЂ” only new folders are created
4. Run `create_env_agent` to regenerate `conda.yml` for new steps
